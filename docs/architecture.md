# ArLABS Control Platform Architecture & Offline-First Protocol

This document defines system integrations, database RLS boundary scopes, service worker lifecycles, and Android client synchronization procedures.

---

## 📱 Offline-First Strategy Protocol

> [!IMPORTANT]
> **CRITICAL ARCHITECTURE NOTE FOR BASE APK:**
> The application must follow a strict Offline-First pattern. All core operational logic, features, and lifetime license validation states must run locally using an encrypted local DB (SQLite/Room). Network calls to Supabase should only happen as non-blocking background synchronizations when Android NetworkCapabilities detect a valid connection. Do not freeze the UI, do not loop loading spinners, and do not block user features for lack of internet.

---

## 🛠 Platform Mechanics

1. **Licensing Control**:
   - The license lookup and activation codes are generated by the Admin PWA console.
   - Keys are downloaded locally by client device APK packages when online and cached securely inside SQLite database schemas.
   - Activations remain fully valid offline.

2. **OTA Update Flow**:
   - Physical `.apk` binaries are hosted on Cloudflare Storage buckets.
   - Supabase tables record download metadata.
   - Devices opportunistically sync database check-ins and pull latest APK updates in the background.

3. **Push Notifications Broadcast Queue**:
   - Firebase Cloud Messaging (FCM) acts as the queue.
   - Messages are queued safely on FCM servers and delivered to devices as soon as they reconnect to cellular or Wi-Fi networks.
